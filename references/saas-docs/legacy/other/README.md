# 📚 Founders Direct Modular - ドキュメント

このディレクトリには、Founders Direct Modular プロジェクトの技術ドキュメントが格納されています。

**Last Updated:** 2025-11-16

---

## 📖 現役ドキュメント

プロジェクトの開発・運用で継続的に参照・更新されるドキュメントです。

### 🔐 [SECURITY.md](./SECURITY.md)

**セキュリティポリシーと実装ガイド**

- **目的**: セキュリティ対策の標準化と脆弱性対応手順の定義
- **対象読者**: 開発者、セキュリティ担当者、運用チーム
- **主な内容**:
  - 認証・認可の実装方法（Google OAuth 2.0）
  - API セキュリティ（レート制限、XSS対策、CSRF対策、SQL対策）
  - データ保護（RLS、暗号化、環境変数管理）
  - 監査とロギング
  - インシデント対応手順

**更新頻度**: 新機能追加時、セキュリティポリシー変更時

---

### 🔒 [RLS-POLICY-GUIDE.md](./RLS-POLICY-GUIDE.md)

**Row Level Security (RLS) 設定ガイド**

- **目的**: データベースレベルのアクセス制御の実装と運用
- **対象読者**: バックエンド開発者、データベース管理者
- **主な内容**:
  - RLS ポリシーの設計と実装
  - テーブル別ポリシー定義（users, workspaces, workspace_members, workspace_data, audit_logs）
  - マイグレーション手順
  - テスト手順（SQL テスト、E2E テスト）
  - トラブルシューティング

**更新頻度**: 新しいテーブル追加時、RLS ポリシー変更時

## 📝 ドキュメント管理ポリシー

### 更新ルール

1. **現役ドキュメント**:
   - 実装と不一致が発生した場合は即座に更新
   - 新機能追加時は関連セクションを追加
   - 変更履歴を末尾に記録

2. **レガシードキュメント**:
   - 原則として更新しない（歴史的資料として保管）
   - ファイル先頭に「このドキュメントはアーカイブされています」と明記

### レガシー化の基準

以下の条件を満たすドキュメントはレガシーに移動します:

- Phase 完了により「設計書」としての役割が終了
- 実装が安定し、将来変更される可能性が低い
- 内容が他のドキュメント（SECURITY.md等）に統合されている
